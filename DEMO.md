# Defense Demo Walkthrough

A step-by-step script for demonstrating the Todo REST API to your lecturer.
Follow it top to bottom. The database starts **empty**, so the migration step
is a real, live "watch the table get created" moment.

> Run everything from inside the `todo/` folder:
> ```powershell
> cd projects\todo
> ```

---

## Part A — Migrations (show the schema being built)

### 1. Prove the `todos` table does NOT exist yet
```powershell
python check_table.py
```
Expected output:
```
todos table does NOT exist  ->  run:  flask db upgrade
```
**Say:** "The database is empty — there is no `todos` table yet."

### 2. Open and explain the migration file
Open: `migrations/versions/855e25af4020_create_todos_table.py`

**Say:** "Flask-Migrate **autogenerated** this by comparing my `Todo` model
(`app/models.py`) against the database. `upgrade()` creates the table,
`downgrade()` removes it — so every schema change is reversible and version-controlled."
Point out that each `sa.Column(...)` matches a column in the model.

### 3. Apply the migration LIVE
```powershell
flask db upgrade
```
Expected output (last line):
```
INFO  [alembic.runtime.migration] Running upgrade  -> 855e25af4020, create todos table
```
**Say:** "This just applied the migration to MySQL and created the table."

### 4. Prove the table now exists
```powershell
python check_table.py
```
Expected output:
```
todos table EXISTS
```

### 5. (Bonus) Show migrations are reversible
```powershell
flask db downgrade     # removes the table
flask db upgrade       # brings it back
```
**Say:** "Migrations can roll backward and forward — this is why teams use them
instead of editing the database by hand."

### 6. (Optional showpiece) Change the model and migrate it LIVE
This proves migrations handle **changes to an existing table**, not just creating
it. Make sure the table already exists first (`flask db upgrade`).

**a) Edit `app/models.py`** — add this column in the columns section:
```python
    # Optional priority label. nullable=True + a default lets it be added to a
    # table that already has rows without breaking them.
    priority = db.Column(db.String(20), nullable=True, default="medium",
                         server_default="medium")
```
and add it to `to_dict()` so the API returns it:
```python
            "completed": self.completed,
            "priority": self.priority,          # <-- add this line
            "created_at": self.created_at.isoformat() if self.created_at else None,
```

**b) Generate a new migration** — Alembic compares the model to the database:
```powershell
flask db migrate -m "add priority column"
```
Expected (key line):
```
INFO  [alembic.autogenerate.compare] Detected added column 'todos.priority'
```

**c) Open the new file** in `migrations/versions/` and point out `op.add_column(...)`.
**Say:** "Alembic autogenerated an `ALTER TABLE` to add just the new column,
without touching existing rows."

**d) Apply it:**
```powershell
flask db upgrade
```

**e) Prove it works** — in Postman, open request **1 (CREATE todo)**, edit the
**Body** to include the new field, and Send:
```json
{
  "title": "Pay rent",
  "priority": "high"
}
```
The response now includes `"priority": "high"`.

> **Why `nullable=True` + a default?** If you add a *required* column with no
> default to a table that already has rows, the migration fails — the existing
> rows would have no value for it. A nullable column with a default is safe.
> Mentioning this shows you understand migrations run against real data.

## Part B — The CRUD REST API in Postman (show it working)

### 1. Start the server
```powershell
python run.py
```
Leave it running. It serves on **http://127.0.0.1:5000**.

### 2. Import the Postman collection (one-time setup)
1. Open Postman.
2. Click **Import** (top-left).
3. Choose the file **`Todo_API.postman_collection.json`** from this project folder
   (drag-and-drop also works).
4. A collection called **"Todo REST API"** appears in the left sidebar with all
   the requests already built.

The base URL is stored as a collection **variable** `{{base_url}}`
(default `http://127.0.0.1:5000`). To change it: click the collection → **Variables**
tab → edit the value. **Say:** "Using a variable means I set the address in one
place instead of editing every request."

### 3. Run the requests top to bottom
Click each request in the sidebar, then press the blue **Send** button. Watch the
response body AND the **status code** (shown top-right of the response panel).

| # | Request | What to point out | Status |
|---|---------|-------------------|--------|
| 0 | Health check | Lists every endpoint — proves the API is up | 200 |
| 1 | CREATE todo | New record returned with a generated `id` | **201** |
| 2 | READ all | The todo you just made appears | 200 |
| 3 | READ one | Fetches a single todo by id | 200 |
| 4 | UPDATE partial (PATCH) | Only `completed` changes — "PATCH = partial" | 200 |
| 5 | UPDATE full (PUT) | Whole record replaced — "PUT = full replace" | 200 |
| 6 | VALIDATION error | No title → `{"error": "title is required"}` | **400** |
| 7 | NOT FOUND | Unknown id → `{"error": "Todo not found"}` | **404** |
| 8 | DELETE | Removes the todo | 200 |

**After request 8 (DELETE):** click request **3 (READ one)** and Send again —
it now returns **404**, proving the record is gone.

> **Note on ids:** requests 3, 4, 5, 8 use id `1`. If your created todo has a
> different id (e.g. you deleted some during practice), edit the number at the end
> of the URL to match the `id` shown in request 1's response.

> **Editing a request body:** click the request → **Body** tab → **raw** (JSON).
> The JSON is already filled in; change the values to demo different data.

## Quick answers to likely questions

| Question | Answer |
|----------|--------|
| What is an ORM? | Object-Relational Mapper — lets me use Python objects (`Todo`) instead of writing raw SQL. `db.session.add/commit/delete` generate the SQL. |
| PUT vs PATCH? | PUT replaces the whole record (title required); PATCH updates only the fields sent. |
| Why an app factory + blueprint? | Separation of concerns, avoids circular imports, and makes the app testable. |
| Where do the status codes come from? | I return them explicitly: 201 create, 200 success, 400 bad input, 404 not found. |
| How are the DB credentials kept safe? | Stored in `.env`, read via python-dotenv; `.env` is gitignored so it's never committed. |
| What does a migration do? | Records a versioned schema change so the database can be built/updated/rolled back consistently. |
