# Todo REST API — Flask + SQLAlchemy

A CRUD REST API for managing a list of todos, built with **Flask**, **SQLAlchemy**
(ORM), **Flask-Migrate** (schema migrations), and **MySQL**.

This README doubles as the defense notes: it explains *what* each part does and
*why* it is structured this way.

---

## 1. What it does

A **Todo** has these fields:

| Field         | Type      | Notes                                    |
|---------------|-----------|------------------------------------------|
| `id`          | integer   | Primary key, auto-generated by MySQL     |
| `title`       | string    | Required, up to 200 characters           |
| `description` | text      | Optional longer notes                    |
| `completed`   | boolean   | Defaults to `false`                      |
| `created_at`  | datetime  | Set automatically when the row is created |

It exposes the full **CRUD** cycle as a JSON REST API.

## 2. Project structure (and why)

```
todo/
├── run.py              # Entry point: builds the app via the factory and runs it
├── app/
│   ├── __init__.py     # create_app() application factory — wires everything together
│   ├── config.py       # Reads .env, builds the DB URI, auto-creates the database
│   ├── extensions.py   # db = SQLAlchemy(), migrate = Migrate()  (unbound instances)
│   ├── models.py       # The Todo model (one class = one table)
│   └── routes.py       # The CRUD endpoints, grouped on a Blueprint
├── .env                # DB credentials (gitignored — never committed)
├── .env.example        # Template showing which variables are needed
├── .flaskenv           # Tells the `flask` CLI where the app is (FLASK_APP=run)
├── requirements.txt    # Exact dependency versions
└── README.md
```

**Why split it up?** Each file has one responsibility, which is the heart of a
"well-structured" app and makes it easy to defend:

- **Application factory (`create_app`)** — builds the app inside a function
  instead of as a global. This enables test copies with different settings and
  avoids circular imports.
- **`extensions.py`** — `db` and `migrate` are created here *unbound* and
  attached later with `init_app()`. `models.py` and the factory both import `db`
  from here, so neither imports the other → no circular import.
- **Blueprint (`routes.py`)** — routes are grouped on a reusable object and
  registered onto the app, keeping route logic separate from app setup.
- **`.env` + `config.py`** — secrets stay out of the code; there is a single
  place to change how we connect to the database.

## 3. The layers (request → response)

```
HTTP client (curl / browser / Postman)
        │
        ▼
routes.py        ← controller: validates input, returns JSON + status codes
        │
        ▼
models.py        ← ORM: Todo objects translated to/from SQL rows
        │
        ▼
MySQL (todos)    ← the actual data
```

## 4. Setup & run

```bash
# 1. (optional) create and activate a virtual environment
python -m venv .venv
.venv\Scripts\activate          # Windows PowerShell:  .venv\Scripts\Activate.ps1

# 2. install dependencies
pip install -r requirements.txt

# 3. create your .env from the template and edit if needed
copy .env.example .env          # macOS/Linux: cp .env.example .env

# 4. create the database tables via a migration
#    (the database itself is auto-created on startup by config.py)
flask db init                   # one time only — creates the migrations/ folder
flask db migrate -m "create todos table"
flask db upgrade

# 5. run the server
python run.py                   # serves on http://127.0.0.1:5000
```

> **Note on migrations:** Flask-Migrate tracks schema changes. After editing
> `models.py`, run `flask db migrate -m "..."` then `flask db upgrade`.
> If you prefer to skip migrations for a quick demo, you can instead create the
> tables directly in a Python shell:
> `from app import create_app; from app.extensions import db;`
> `app = create_app();`
> `app.app_context().push(); db.create_all()`

## 5. API reference

Base URL: `http://127.0.0.1:5000`

| Method   | Endpoint           | Purpose                       | Success |
|----------|--------------------|-------------------------------|---------|
| `GET`    | `/api/todos`       | List all todos (newest first) | 200     |
| `GET`    | `/api/todos/<id>`  | Get one todo                  | 200     |
| `POST`   | `/api/todos`       | Create a todo                 | 201     |
| `PUT`    | `/api/todos/<id>`  | Replace all fields            | 200     |
| `PATCH`  | `/api/todos/<id>`  | Update only the fields sent   | 200     |
| `DELETE` | `/api/todos/<id>`  | Delete a todo                 | 200     |

Error responses: `400` (invalid/missing data), `404` (todo not found),
each as `{"error": "..."}`.

### Example requests (curl)

```bash
# CREATE
curl -X POST http://127.0.0.1:5000/api/todos \
  -H "Content-Type: application/json" \
  -d "{\"title\": \"Buy milk\", \"description\": \"2 litres\"}"

# READ all
curl http://127.0.0.1:5000/api/todos

# READ one
curl http://127.0.0.1:5000/api/todos/1

# UPDATE (partial) — mark as completed
curl -X PATCH http://127.0.0.1:5000/api/todos/1 \
  -H "Content-Type: application/json" \
  -d "{\"completed\": true}"

# UPDATE (full replace)
curl -X PUT http://127.0.0.1:5000/api/todos/1 \
  -H "Content-Type: application/json" \
  -d "{\"title\": \"Buy oat milk\", \"description\": \"1 litre\", \"completed\": false}"

# DELETE
curl -X DELETE http://127.0.0.1:5000/api/todos/1
```

## 6. Likely defense questions

- **What is an ORM?** Object-Relational Mapper — it lets us work with Python
  objects (`Todo`) instead of writing raw SQL. `db.session.add/commit/delete`
  generate the SQL for us.
- **PUT vs PATCH?** PUT replaces the whole resource (title required); PATCH
  updates only the fields you send (e.g. just toggling `completed`).
- **Why a factory and blueprint?** Separation of concerns + avoiding circular
  imports + testability — see section 2.
- **Where do status codes come from?** We return them explicitly: `201` on
  create, `400` on bad input, `404` when a todo id doesn't exist, `200` otherwise.
- **How are secrets handled?** Read from `.env` via python-dotenv; `.env` is
  gitignored so credentials never reach the repository.
